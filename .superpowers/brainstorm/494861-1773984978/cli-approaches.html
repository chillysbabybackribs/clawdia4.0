Three approaches for the calendar CLI + live update

All use SQLite + CLIanything — the difference is how the UI stays live

A

CLI → SQLite → IPC Poll

The CLI writes events to SQLite. The Electron main process polls the DB every few seconds and pushes updates to the renderer via an existing IPC channel. Calendar UI re-renders on each push.


Pros

  • Simple — no file watchers or sockets
  • Polling interval tunable (1–5s)

Cons

  • Slight lag (up to poll interval)
  • Constant background work even when idle
B

CLI → SQLite → File Watcher → IPC Push ★ Recommended

The CLI writes to SQLite. The main process watches the DB file with fs.watch. On change, it reads the new events and pushes them to the renderer instantly via IPC. Zero polling.


Pros

  • Instant updates — no lag
  • Zero overhead when nothing changes
  • Fits cleanly into existing IPC pattern

Cons

  • fs.watch can fire multiple times per write — needs debounce (trivial)
C

CLI → SQLite → Named Pipe / Unix Socket

The CLI writes to SQLite then sends a signal over a Unix socket. The main process listens on the socket and immediately triggers a UI refresh.


Pros

  • Truly instant, no debounce needed
  • Clean separation of concerns

Cons

  • More complex — socket lifecycle, cleanup
  • Overkill for this use case
